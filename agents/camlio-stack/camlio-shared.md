## Naming Conventions

Every block name must be consistent across all files. Use this table as the single source of truth:

| Context | Format | Example |
|---|---|---|
| Sanity schema `name` | camelCase | `heroSection` |
| Sanity schema `title` | Title Case | `Hero Section` |
| TypeScript type (auto-generated) | PascalCase | `HeroSection` |
| React component file/folder | PascalCase | `HeroSection/index.tsx` |
| React component export name | PascalCase + `Block` suffix | `HeroSectionBlock` |
| `BlockRenderer` map key | camelCase (= schema `name`) | `heroSection` |

**Rule:** the `name` in the Sanity schema is the canonical identifier. Everything else derives from it.

## End-to-End Block Contract

A "block" is a self-contained page section managed by editors in Sanity Studio and rendered by a React component in Next.js. Every new block must complete all steps below to be fully wired up.

### Option A ÔÇö Automated (recommended)

```bash
# From repo root ÔÇö name must be PascalCase
node create-block.js HeroSection
```

This scaffolds:
- `studio/src/schemaTypes/objects/HeroSection/index.ts`
- `frontend/app/components/blocks/HeroSection/index.tsx`
- Registers the block in `importSchemaType.ts`
- Registers the block in `importPageBuilderTypes.ts`

Then continue from **Step 4** (regenerate types) and **Step 5** (wire up BlockRenderer).

### Option B ÔÇö Manual

#### Step 1 ÔÇö Create the Sanity schema object

`studio/src/schemaTypes/objects/MyBlock/index.ts` Ôćĺ see `camlio-studio.md` for full schema template.

#### Step 2 ÔÇö Register the schema

`studio/src/schemaTypes/importSchemaType.ts`:

```ts
import { myBlock } from './objects/MyBlock'

export const importSchemaTypes = [
  // ÔÇŽexisting
  myBlock,
]
```

Alternatively add directly to `studio/src/schemaTypes/index.ts` for core blocks.

#### Step 3 ÔÇö Add to the page builder field

`studio/src/schemaTypes/objects/importPageBuilderTypes.ts`:

```ts
export const importPageBuilderTypes = [
  // ÔÇŽexisting
  { type: 'myBlock' },
]
```

#### Step 4 ÔÇö Regenerate TypeScript types

```bash
# From repo root:
npm run predev
# Or from /frontend:
npm run sanity:typegen
```

This updates `frontend/sanity.types.ts` with the new `MyBlock` type. **Always run this after any schema change.**

#### Step 5 ÔÇö Create the React component

`frontend/app/components/blocks/MyBlock/index.tsx` Ôćĺ see `camlio-frontend.md` for the full component template.

#### Step 6 ÔÇö Register in BlockRenderer

`frontend/app/components/BlockRenderer.tsx`:

```ts
import MyBlockBlock from '@/app/components/blocks/MyBlock'

const Blocks = {
  // ÔÇŽexisting
  myBlock: MyBlockBlock,
}
```

#### Step 7 ÔÇö (Optional) Add Studio thumbnail

Place a PNG at `studio/static/page-builder-thumbnails/myBlock.png`.  
This image is shown in the Studio "Add block" grid menu.

### Registration Checklist

| Step | File | Action |
|---|---|---|
| 1 | `studio/src/schemaTypes/objects/<BlockName>/index.ts` | Create schema object |
| 2 | `studio/src/schemaTypes/importSchemaType.ts` | Import + add to array |
| 3 | `studio/src/schemaTypes/objects/importPageBuilderTypes.ts` | Add `{ type: 'myBlock' }` |
| 4 | Run `npm run predev` from root | Regenerate `sanity.types.ts` |
| 5 | `frontend/app/components/blocks/<BlockName>/index.tsx` | Create React component |
| 6 | `frontend/app/components/BlockRenderer.tsx` | Add to `Blocks` map |
| 7 | `studio/static/page-builder-thumbnails/myBlock.png` | (Optional) Studio thumbnail |

## TypeScript Type Generation

Types are auto-generated by `sanity typegen` from the schema. The flow is:

```
/studio schema  Ôćĺ  sanity.schema.json (root)  Ôćĺ  frontend/sanity.types.ts
```

**Never edit `sanity.types.ts` or `sanity.schema.json` manually.**

### Importing generated types

```ts
// Block-level types (auto-generated)
import { MyBlock, Page, Hero } from '@/sanity.types'

// Utility types defined in /frontend/sanity/lib/types.ts
import { PageBuilderSection, ExtractPageBuilderType } from '@/sanity/lib/types'

// Extract a specific block type from the pageBuilder union
type HeroBlock = ExtractPageBuilderType<'hero'>
```

### Key files

| File | Purpose |
|---|---|
| `frontend/sanity.types.ts` | Auto-generated ÔÇö all Sanity types |
| `frontend/sanity/lib/types.ts` | Hand-written utilities: `PageBuilderSection`, `ExtractPageBuilderType`, `DereferencedLink` |
| `frontend/sanity/lib/api.ts` | Exports env vars: `projectId`, `dataset`, `apiVersion`, `studioUrl` |
| `frontend/sanity/lib/client.ts` | `createClient()` with stega support |
| `frontend/sanity/lib/live.ts` | `sanityFetch()` + `SanityLive` for Live Content API |
| `frontend/sanity/lib/queries.ts` | All GROQ queries using `defineQuery` |
| `frontend/sanity/lib/service.ts` | Data fetching helpers: `getPageBySlug`, `getTemplates`, etc. |
| `frontend/sanity/lib/utils.ts` | `urlForImage`, `linkResolver`, `dataAttr`, `cn`, `formatPrice` |

## GROQ Query Patterns

All queries live in `frontend/sanity/lib/queries.ts` and use `defineQuery` from `next-sanity` for type inference.

### Basic structure

```ts
import { defineQuery } from 'next-sanity'

export const getPageQuery = defineQuery(`
  *[_type == 'page' && slug.current == $slug][0]{
    _id, _type, name,
    "slug": slug.current,
    "pageBuilder": pageBuilder[]{ ... },
    ...
  }
`)
```

- `*[filter]` ÔÇö filter all documents
- `[0]` ÔÇö take first result (single document)
- `{ ÔÇŽ }` ÔÇö projection (select/rename fields)
- `...` ÔÇö spread all remaining fields

### Projections & renaming

```groq
{
  _id,
  _type,
  "slug": slug.current,              // rename slug object Ôćĺ string
  "author": author->{ name },        // dereference + project
  "date": coalesce(date, _updatedAt) // fallback value
}
```

### Dereferencing references

```groq
// Single reference
"author": author->{ firstName, lastName, picture }

// Array of references
"tags": tags[]->{ _id, tagName }

// Nested dereference
"category": category->{ title, "parentTitle": parent->title }
```

### Conditional projections per block type

```groq
"pageBuilder": pageBuilder[]{
  ...,
  _type == "callToAction" => {
    ...,
    button {
      ...,
      link {
        ...,
        _type == "link" => {
          "page": page->slug.current,
          "post": post->slug.current
        }
      }
    }
  },
  _type == "infoSection" => {
    content[]{
      ...,
      markDefs[]{
        ...,
        _type == "link" => {
          "page": page->slug.current
        }
      }
    }
  }
}
```

### Image fields

```groq
// Basic image
image { ..., asset->{ _id, url } }

// With dimensions
thumbnail {
  ...,
  asset->{
    _id, url,
    metadata { dimensions { width, height } }
  }
}
```

### SEO fields

```groq
seo{
  _type, metaTitle, metaDescription, seoKeywords,
  "nofollowAttributes": coalesce(nofollowAttributes, null),
  openGraph{ _type, siteName, url, description, title, image{ ... } },
  twitter{ _type, site, creator, cardType, handle },
  additionalMetaTags[]{ _type, metaAttributes[]{ ... } }
}
```

### Ordering & pagination

```groq
// Order by date descending
*[_type == "template"] | order(_createdAt desc)

// Paginate (slice)
*[_type == "template"] | order(_createdAt desc) [0...10]

// Dynamic limit via parameter
*[_type == "template"] | order(_createdAt desc) [0...$limit]
```

### Filtering with parameters

```groq
// By slug
*[_type == "page" && slug.current == $slug][0]

// Conditional search
*[
  _type == "template" &&
  defined(slug.current) &&
  ($search == null || title match $search + "*" || description match $search + "*")
]
```

### Sitemap query pattern

```groq
*[(_type == "page" || _type == "template") && defined(slug.current)]
| order(_type asc) {
  "slug": slug.current,
  _type, _updatedAt
}
```

**Data flow per page request:**

1. Request arrives at `[slug]/page.tsx` (or root `page.tsx` for home).
2. `sanityFetch({ query: getPageQuery, params: { slug } })` calls Sanity Content Lake.
3. The `page.pageBuilder[]` array is returned as a typed union (`PageBuilderSection`).
4. `<PageBuilder>` wraps it with `useOptimistic` for real-time Studio edits.
5. For each block, `<BlockRenderer>` maps `block._type` Ôćĺ React component Ôćĺ renders HTML.

## Programmatic Content Scripts (Migrations)

After blocks are implemented and wired up, create a script to seed or migrate content into Sanity programmatically. This avoids manual data entry in the Studio and is useful for new page setup, content migrations, and test data.

### Script location & naming

Scripts live in `scripts/` at the repo root. Follow the naming pattern of existing scripts:

```
scripts/
ÔöťÔöÇÔöÇ migrate-homepage.mjs
ÔöťÔöÇÔöÇ migrate-globals.mjs
ÔööÔöÇÔöÇ seed-<page-slug>.mjs   ÔćÉ your new script
```

### Pattern

```js
// scripts/seed-our-story.mjs
import { createClient } from '@sanity/client'

const client = createClient({
  projectId: process.env.SANITY_STUDIO_PROJECT_ID,
  dataset: process.env.SANITY_STUDIO_DATASET,
  token: process.env.SANITY_API_WRITE_TOKEN,
  apiVersion: '2025-09-25',
  useCdn: false,
})

await client.createOrReplace({
  _id: 'our-story-page',
  _type: 'page',
  name: 'Our Story',
  slug: { _type: 'slug', current: 'our-story' },
  pageBuilder: [
    {
      _type: 'hero',
      _key: 'hero-1',
      headline: 'Our Story',
    },
    {
      _type: 'couple',
      _key: 'couple-1',
      heading: 'Meet the Couple',
    },
  ],
})

console.log('Done.')
```

Run from the repo root:

```bash
node --env-file=studio/.env scripts/seed-our-story.mjs
```

### Key rules

| Rule | Detail |
|---|---|
| `_key` on every array item | Required for Sanity array items ÔÇö use a descriptive slug or `crypto.randomUUID()` |
| `createOrReplace` vs `create` | Use `createOrReplace` for idempotent re-runs; `create` fails if the document already exists |
| Image fields | Upload first via `client.assets.upload('image', fs.createReadStream(path))`, then reference: `{ _type: 'image', asset: { _type: 'reference', _ref: asset._id } }` |
| Write token | Script needs `SANITY_API_WRITE_TOKEN` with `editor` role ÔÇö never commit the token |
| Follow existing scripts | Check `scripts/migrate-homepage.mjs` and `scripts/migrate-globals.mjs` for the established pattern |
